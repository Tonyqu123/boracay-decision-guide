# Boracay Case

长滩岛是本项目的第一个 case。

它足够小，适合做深；也足够复杂，能暴露旅行决策里的典型问题。

---

## 1. 为什么选长滩岛

长滩岛不是一个信息稀缺目的地。

相反，它的问题是：

- 信息太多。
- 中文信息和英文信息割裂。
- 平台信息和本地信息割裂。
- 酒店、交通、活动高度依赖位置和季节。
- 很多选择看起来类似，但实际体验差别很大。

这让它适合作为“旅行决策层”的测试场景。

---

## 2. 早期判断范围

### 2.1 区域

重点理解：

- Station 1
- Station 2
- Station 3
- Bulabog Beach
- Diniwid
- Yapak / 北部区域

判断问题：

- 想安静住哪里？
- 想方便吃饭和夜生活住哪里？
- 带老人孩子是否适合住偏远酒店？
- 住海边和不住海边的真实差异是什么？

### 2.2 交通

重点理解：

- Caticlan 机场到酒店。
- Kalibo 机场到酒店。
- 车船联运。
- 酒店接送。
- 岛上 tricycle。

判断问题：

- 哪种路径省心？
- 哪种路径便宜但消耗高？
- 晚到、带孩子、带大行李时风险是什么？
- 平台接送和酒店接送如何比较？

### 2.3 酒店

重点理解：

- 区域对体验的影响。
- 中文平台评价和 Google Maps 评价的差异。
- 海景、沙滩距离、安静程度、设施新旧之间的取舍。
- “网红酒店”和“稳定履约酒店”的区别。

判断问题：

- 哪类用户适合 Station 1？
- 哪类用户适合 Station 2？
- 哪类用户不该为了便宜住太偏？
- 如何从差评判断酒店是否真的有问题？

### 2.4 潜水和海上活动

重点理解：

- 体验潜水
- 浮潜
- 考证
- fun dive
- 跳岛和海上活动

判断问题：

- 不会游泳能不能体验潜水？
- 哪些人不适合体验潜水？
- 潜店评价里哪些差评是高风险？
- 便宜套餐可能牺牲什么？

## 3. 信息源

长滩岛 case 需要交叉使用多类信息源：

- Google Maps：地点、评分、近期评论、照片。
- Agoda / Booking：酒店价格、房型、住客反馈。
- 携程 / 飞猪：中文用户反馈、套餐描述。
- Klook / GetYourGuide：活动商品、取消政策、接送说明。
- TripAdvisor：英文游客长期评价。
- 小红书 / 马蜂窝：中文语境体验和踩坑。
- 商家官网 / Facebook：本地商家一手信息。
- 用户访谈 / 真实行程反馈：补足平台盲区。

单一平台只能作为线索，不能直接作为结论。

## 6. 判断目标

长滩岛 case 的目标不是告诉用户：

> 长滩岛怎么玩最完美？

而是帮助用户回答：

> 在我的预算、时间、同行人、风险偏好和语言能力下，哪些选择更不容易后悔？
